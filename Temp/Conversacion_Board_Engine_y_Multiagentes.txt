🚀 Optimizando la Arquitectura Multiagente
1. El Game Designer como "Single Source of Truth" (SSOT)
Introducir el rol de Game Designer para serializar el reglamento en datos (JSON/YAML) antes de que el agente Shared toque una sola línea de código es el movimiento más brillante de la propuesta.

Beneficio: Evitas que las reglas queden embebidas fuertemente en el código TypeScript. Si mañana quieres cambiar el coste de una localización o balancear una carta, solo tocas el archivo de datos.

Flujo: .ai/dune/rules.md (Texto humano) ➡️ Game Designer Agent ➡️ packages/shared/src/data/ (JSON estructurado) ➡️ Shared Agent (Tipados y validadores estáticos).

2. El Agente QA como "Muro de Contención"
Para un juego con tanta interdependencia de reglas, el agente QA no debe limitarse a escribir tests unitarios aislados. Su mayor valor aquí será la simulación:

Debería implementar un script de simulación (fuzzing) donde dos agentes Bot jueguen partidas aleatorias completas a máxima velocidad.

Esto es crucial para detectar deadlocks (ej. jugadores bloqueados sin poder hacer acciones válidas) o estados corruptos en el servidor antes de que integres el cliente.

3. La barrera del contexto en .ai/
La idea de separar los contextos en archivos .md específicos es vital para Claude Code o cualquier agente local. Si el agente Client lee todo el reglamento de combate, desperdicias tokens y diluyes su atención.

El archivo client.md solo debe decirle: "Renderiza lo que te venga en el GameState.combat. Si hay un conflicto, muestra el componente X. No calcules quién gana."

🔍 Los 3 Factores que hacen a Dune más complejo de programar
1. La pesadilla de la Información Oculta y el Estado Filtrado (Server/Shared)
En Brass, el 90% del tablero es público; solo necesitas ocultar qué cartas tiene cada jugador en su mano.
En Dune, el servidor tiene que lidiar con tres niveles de opacidad:

El mazo de Cartas de Intriga: El servidor debe validar si un jugador puede jugar una carta de intriga en la fase de combate sin que los demás sepan qué carta es hasta que se ejecuta.

El mazo de Conflictos: Hay que gestionar revelaciones programadas.

El agente Bot no puede hacer trampas; hay que programar el servidor para que "filtre" el estado global y envíe al bot (y al cliente) exclusivamente lo que su vista legal le permite conocer.

2. La Fase de Combate y la Asincronía (Server)
Brass es estrictamente secuencial: Turno del Jugador A (Acción 1, Acción 2), luego Jugador B. Es un flujo lineal fácil de implementar con un patrón de comandos clásico.
Dune rompe la linealidad en la Fase de Combate:

Hay una ronda de juego de cartas de intriga de combate donde los jugadores pasan o juegan hasta que todos pasan consecutivamente.

Gestionar ese sub-bucle de turnos asíncrono, donde el estado de la fuerza militar fluctúa constantemente antes de la resolución final, introduce muchas más condiciones de carrera potenciales en los WebSockets.

3. Efectos de cartas e Inyección de Reglas (Game Designer / Shared)
En Brass, las acciones son sota, caballo y rey (Construir, Red, Desarrollar, Vender, Préstamo, Scout). Las reglas son rígidas y matemáticas.
En Dune, las cartas de Imperio e Intriga inyectan excepciones a las reglas constantemente:

"Si tienes la alianza de la Cofradía, esta carta te da +2 de especia."

"Al revelar, si tienes otra carta con el símbolo X, activa Y."
Esto obliga al agente Game Designer y al Shared a crear un motor de efectos basado en eventos mucho más robusto para que las cartas puedan mutar el flujo normal de una acción.

🛠️ ¿Dónde gana Brass: Birmingham?
El único punto donde Brass aplasta a Dune en complejidad es en el módulo de rutas y conectividad (Client y Server):

Validar si una localización está conectada a una fuente de carbón requiere implementar algoritmos de búsqueda en grafos (como BFS o Dijkstra) sobre el estado dinámico del mapa (canales y vías de tren construidas).

Renderizar visualmente en PixiJS las conexiones compartidas y las losetas volteadas de forma limpia requiere más trabajo de arquitectura visual que el tablero de Dune, que es fundamentalmente una colección de botones (localizaciones) fijos.

📐 El Modelo Unificado: El Espacio de Juego como "Nodo"
Para el agente Game Designer y el motor de renderizado (Client), un espacio del tablero de Clanes de Caledonia, una localización de Dune, o una ciudad de Brass son exactamente lo mismo: un Nodo con propiedades y restricciones.

Si defines un esquema JSON genérico para estos nodos, te sirve para todo:

JSON
{
  "id": "arrakeen_dune",
  "type": "worker_slot",
  "coordinates": { "x": 450, "y": 320 },
  "requirements": {
    "space_type": "landsraad",
    "cost": { "persuasion": 2 }
  },
  "rewards": [
    { "type": "gain_resource", "resource": "troop", "amount": 2 },
    { "type": "draw_card", "deck": "imperium", "amount": 1 }

}
Cómo se adapta este mismo esquema a cada juego:
Dune: Imperium: El nodo es una localización (ej. Arrakeen). Su requirement es el icono de la carta (Landsraad) y su reward es dar tropas/agua.

Clanes de Caledonia: El nodo es una casilla de mapa (un hexágono de montaña). Su requirement es que esté libre, que pagues el coste del terreno y que esté adyacente a tus otras estructuras (aquí entra el concepto de grafo). Su reward es colocar tu loseta de industria.

Brass: Birmingham: El nodo es un slot de industria en una ciudad (ej. el slot de fábrica de algodón en Birmingham). Sus requirements son la era actual (Canal/Rail) y tener la carta de la ciudad o de la industria.

📐 Cómo estructurar este módulo en Board Engine
Tu enfoque de éxito de Brass se puede replicar en Board Engine de forma ultra-limpia en tres piezas:

En packages/shared (o core): Defines el tipo BoardElement (ID, x, y, ancho, alto, tipo).

En packages/client (Render): Creas un componente contenedor (ej. <BoardLayoutEditor>) que envuelve el lienzo del tablero. Si la variable de edición está activa, intercepta los inputs, hace los elementos draggable en PixiJS y muestra un botón flotante de "Guardar Configuración".

En packages/server (Dev API): Un único endpoint de desarrollo (protegido para que no se suba a producción) que recibe el JSON con las nuevas coordenadas y usa fs.writeFileSync para actualizar el archivo del juego correspondiente (/games/dune/layout.json).

De esta forma, cuando pases de Dune a Clanes de Caledonia, el sistema de posicionamiento ya estará integrado en el motor. Solo tendrás que cambiar la imagen de fondo del tablero y la lista de IDs de los nodos.

# Board Engine - Arquitectura del Sistema

Eres un agente de IA que trabaja en un monorepo TypeScript estructurado mediante workspaces.

## Estructura de Paquetes
- `packages/shared`: Tipos comunes, esquemas de datos (Zod/JSON), lógica de estado inmutable (reducers) y utilidades puras. No dependes de Node.js ni del DOM.
- `packages/server`: Servidor de juego (Node.js/Express/WebSockets). Valida comandos, gestiona la persistencia, oculta información y maneja las sesiones de juego.
- `packages/client`: Frontend de la aplicación (React, PixiJS, Tailwind CSS). Renderizado visual, HUD y captura de inputs del usuario. Nunca valida reglas de juego de forma definitiva; asume lo que dicte el servidor.

## Flujo de Datos
1. El Cliente emite una acción/comando vía WebSocket.
2. El Servidor recibe la acción, valida si es legal usando las reglas compartidas (`shared`).
3. Si es válida, el Servidor aplica el reducer para mutar el estado global, filtra la información oculta para cada jugador y propaga el nuevo estado.

# Flujo de Trabajo Multiagente

Para cualquier funcionalidad (como el Layout Editor o mecánicas de Dune), el desarrollo debe seguir estrictamente este orden secuencial de agentes:

1. **Planner**: Analiza el requisito, lee las reglas y divide el trabajo en subtareas atómicas sin picar código. Crea las tareas en un archivo temporal de progreso.
2. **Game Designer**: Si la tarea requiere datos (cartas, localizaciones, mapas), los modela en archivos JSON/YAML estáticos.
3. **Shared Agent**: Implementa o actualiza las interfaces TypeScript, tipos y DTOs basados en el diseño.
4. **Server Agent**: Desarrolla endpoints, controladores y lógica de estado.
5. **Client Agent**: Desarrolla la interfaz visual, componentes de React o contenedores de PixiJS.
6. **QA Agent**: Escribe los tests unitarios y de integración para cerrar la tarea.

## Ley de Restricción de Contexto
Cada agente técnico (Shared, Server, Client, Bot) tiene estrictamente PROHIBIDO modificar archivos fuera de su scope (`packages/[scope]`). Si un Agente de Cliente necesita un cambio en un tipo de datos, debe pausar su ejecución y solicitar que el Agente de Shared actúe primero.

Por qué van en esa carpeta .ai?

La elección de la carpeta .ai/ en la raíz del proyecto es una decisión puramente de arquitectura de desarrollo e ingeniería de prompts. No es un estándar impuesto por Node.js o TypeScript, sino una convención excelente por tres motivos técnicos de peso:

1. El estándar "Dot-Folder" (.something)
En desarrollo, las carpetas que empiezan por punto (como .github/, .vscode/ o .husky/) se utilizan para configurar herramientas que automatizan o asisten en tu flujo de trabajo, pero que no forman parte del código que se ejecuta en producción. Guardar ahí tus prompts y reglas del sistema deja claro que es infraestructura de desarrollo.

2. Facilidad para alimentar el Contexto de la IA
Herramientas como Claude Code, Copilot o cursores de IA indexan tu proyecto para entenderlo. Al tener todo centralizado en una única carpeta, puedes darle directrices ultra-precisas en tu prompt inicial:

"Lee siempre el archivo .ai/system/architecture.md antes de proponer cambios."
o bien
"Si actúas como Server Agent, inicializa tu contexto cargando .ai/prompts/server.md".

Es una forma limpia de "inyectarle el chip" de su rol de manera inmediata.

3. Escondido de la vista de producción (.gitignore)
Estos archivos de texto son pesados en tokens y solo te interesan mientras estás programando. Al estar bajo una carpeta común, puedes gestionar perfectamente su ciclo de vida. Por ejemplo, querrás subirlos a tu repositorio Git para que si trabajas desde otro equipo tus agentes sigan teniendo sus instrucciones, pero nunca se empaquetarán en el dist/ final del servidor ni del cliente.

# Rol: Game Designer Agent (Especialista en Datos de Juego)

Eres el diseñador de sistemas encargado de traducir reglamentos de juegos de mesa a estructuras de datos puras (JSON/YAML) para Board Engine. Tu objetivo es separar por completo las reglas conceptuales de la implementación en código.

## Tu Enfoque: Colocación de Trabajadores Pura
Trabajarás con juegos donde la mecánica central consiste en que un jugador reclama un espacio (Slot/Location) bloqueándolo para los demás, paga un coste y recibe un beneficio inmediato o diferido.

## Tus Responsabilidades:
1. **Modelado de Espacios (Locations):** Definir cada casilla del tablero como un objeto con IDs únicos.
2. **Estandarización de Requisitos:** Traducir las restricciones de entrada (ej: "Cuesta 2 monedas", "Solo un trabajador por espacio") a un DSL (Lenguaje Específico de Dominio) en JSON.
3. **Estandarización de Recompensas/Efectos:** Traducir los beneficios (ej: "Gana 3 de madera", "Roba 2 cartas") a acciones atómicas que el motor pueda procesar.

## Restricciones Estrictas:
- NUNCA escribas código TypeScript, React ni lógica de servidor. Tu salida debe ser EXCLUSIVAMENTE esquemas de datos (JSON/YAML) o documentación de reglas en Markdown.
- Mantén las acciones de recompensa lo más atómicas posible para que sean reutilizables (ej: en lugar de `efecto: "dar_recursos_iniciales"`, usa un array de `[{ "action": "GAIN_RESOURCE", "type": "wood", "amount": 2 }]`).

## Ejemplo de Salida Esperada para una Localización:
```json
{
  "id": "blacksmith_shop",
  "name": "Armería",
  "category": "board_location",
  "worker_capacity": 1,
  "requirements": {
    "cost": { "gold": 1 }
  },
  "effects": [
    { "type": "GAIN_RESOURCE", "resource": "iron", "amount": 2 },
    { "type": "GAIN_VP", "amount": 1 }
  ]
}
